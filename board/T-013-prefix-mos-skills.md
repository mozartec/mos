# T-013 — Prefix first-party skills with the mos- namespace

The two first-party skills install under bare names (`next-card`, `ship-card`), so in a
host repo that also uses other skills they read as generic and can collide. Namespacing
them `mos-*` marks provenance and keeps them distinct wherever mos is adopted (the
portability goal behind F-014). This is a rename only — no behaviour change.

## Outcome

- `skills/next-card` → `skills/mos-next-card` and `skills/ship-card` →
  `skills/mos-ship-card`, with matching `name:` frontmatter; invocation becomes
  `/mos-next-card` and `/mos-ship-card`.
- The GitHub agent `ship-card-agent` → `mos-ship-card-agent`
  ([`.github/agents/ship-card-agent.agent.md`](../.github/agents/ship-card-agent.agent.md)),
  loading the renamed `mos-ship-card` skill.
- Installed copies under `.agents/skills/` and `skills-lock.json` **regenerated through
  the skills CLI** (`npx skills add`, local-path form) — not hand-edited; the
  `computedHash` is CLI-owned (T-009).
- The naming rule recorded once: **first-party mos skills carry the `mos-` prefix**
  ([`skills/README.md`](../skills/README.md)); the future refine-batch skill (F-027)
  is authored as `mos-refine-batch`.
- Only the `mozartec/mos`-sourced skills are renamed. The third-party installs
  (`angular-developer`, `angular-new-app`, `daisyui`, `turborepo`) keep their upstream
  names — they aren't ours to namespace.

## Context — read before starting

- T-009 — the CLI-managed install model: installed copies + `skills-lock.json` come from
  `npx skills add`, never hand edits. This task's whole reason for being a task (not a
  hand sweep) is that the lockfile hash isn't reproducible by hand.
- [`skills/README.md`](../skills/README.md) — install instructions and the naming rule
  to update.
- [`AGENTS.md`](../AGENTS.md) §Agent skills — where this repo references the two skills.
- [`.github/agents/ship-card-agent.agent.md`](../.github/agents/ship-card-agent.agent.md)
  and [`.github/labeler.yml`](../.github/labeler.yml) — the agent and any path globs that
  name the skills.

## Constraints (must honor)

- **CLI-only for installed copies + lockfile** (T-009): regenerate them with the skills
  CLI; never hand-edit `.agents/skills/**` or `skills-lock.json`.
- **Live references only.** Update `AGENTS.md`, `skills/README.md`, the affected docs
  (`07-ROADMAP`, `09-CONVENTIONS`, `12-ADOPTING`, `11-RELEASING`, and `08-DECISIONS` if
  it names a skill), and the live board cards (F-014, F-025, F-027). **Do not rewrite
  `CHANGELOG.md`** or other historical records — they correctly name what shipped then.
- **Rename only, no behaviour change.** The skills stay vault-agnostic; scripts and
  evals move with their skill unchanged.
- **Third-party skills untouched.**
- Bump each renamed skill's version per [`skills/README.md`](../skills/README.md) so the
  rename is a visible release of the skill.

## Plan

1. `git mv skills/next-card skills/mos-next-card` and `skills/ship-card
   skills/mos-ship-card`; set `name:` in each `SKILL.md`; fix any self-references in the
   skill body, scripts, and evals.
2. `git mv` the agent to `mos-ship-card-agent.agent.md`; update its `name:` and the
   `mos-ship-card` skill reference.
3. Reinstall via the skills CLI (local-path form) to regenerate `.agents/skills/mos-*`
   and `skills-lock.json`; confirm the new entries are keyed `mos-next-card` /
   `mos-ship-card`, sourced `mozartec/mos`, with fresh hashes.
4. Sweep live references (step list in Constraints); add the `mos-` naming rule to
   `skills/README.md`; leave `CHANGELOG.md` alone.
5. Update `.github/labeler.yml` and `.claude/settings.local.json` skill paths if they
   name the old skill folders.
6. `bun run validate`; verify `/mos-next-card` and `/mos-ship-card` resolve and that
   `npx skills add` against the repo discovers the prefixed names.
