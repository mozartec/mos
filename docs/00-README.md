# Documentation index

This folder is the source of truth for what mos is and how it's built. It's written for
two audiences: people evaluating the project, and contributors (human or AI) who need to
understand the design before changing it.

## Reading order

1. [`01-VISION.md`](01-VISION.md) — what we're building and for whom
2. [`02-CONCEPTS.md`](02-CONCEPTS.md) — the core mental model
3. [`05-VAULT_SPEC.md`](05-VAULT_SPEC.md) — the vault format (the data contract)
4. [`06-MVP.md`](06-MVP.md) — the first version's scope
5. [`03-ARCHITECTURE.md`](03-ARCHITECTURE.md) — how the app is structured
6. [`04-TECH_STACK.md`](04-TECH_STACK.md) — the stack and why
7. [`08-DECISIONS.md`](08-DECISIONS.md) — the decisions behind the above
8. [`07-ROADMAP.md`](07-ROADMAP.md) — where it goes after MVP
9. [`09-CONVENTIONS.md`](09-CONVENTIONS.md) — how to write specs, cards, ADRs
10. [`10-GLOSSARY.md`](10-GLOSSARY.md) — terminology
11. [`11-RELEASING.md`](11-RELEASING.md) — branching, commits, versioning, pipelines

See also the project [`CHANGELOG.md`](../CHANGELOG.md) at the repo root.

## Principle

These docs describe *intent*. Code (once it exists) describes *truth*. When they
disagree, code wins and a docs update is required to bring intent back in sync.
